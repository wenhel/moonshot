# API Cost & Latency 记录

## 2026-04-20 Seg30 Movement 描述实验

**模型**: Gemini 2.5 Flash
**价格**: $0.15/M input tokens, $0.60/M output tokens
**Segment**: seg30 (900s-930s, 30秒片段)

### 三种输入方式对比

| 条件 | 输入 | 延迟 | 输入 tokens | 输出 tokens | 单段成本 | 31段总成本 | 质量 |
|---|---|---|---|---|---|---|---|
| A: 6帧 | 6 张 keyframe (874KB) | 2.2s | ~1,565 | ~110 | $0.00030 | ~$0.009 | 0/5 |
| B: 30帧 | 30 张 keyframe (6.6MB) | 3.5s | ~7,757 | ~156 | $0.00126 | ~$0.039 | 2/5 |
| C: 视频 | 30s mp4 (13MB) | 17.6s | ~7,865 | ~508 | $0.00148 | ~$0.046 | 4.5/5 |

### Token 估算依据

- 图片: ~258 tokens/张 (Gemini 标准, 768px resize)
- 视频: ~263 tokens/秒 (Gemini 标准)
- 文本: ~1.3 tokens/word

### 延迟分解 (条件 C)

- 视频上传: 1.4s
- 服务端处理: 3.3s
- 生成: 12.9s
- 总计: 17.6s

### 关键发现

1. **B 和 C 输入 token 几乎相同** (~7.8k), 但 C 的输出信息量 3x (508 vs 156 tokens)
2. **C 成本仅比 B 高 17%** ($0.00148 vs $0.00126), 但质量差 2.5 倍 (4.5 vs 2)
3. **全视频 31 段总成本不到 $0.05**, 三种方式成本差距可忽略
4. **C 延迟最高** (17.6s vs 3.5s), 但如果不需要实时, 这不是问题
5. **A 最便宜但完全不可用** — 6帧太少导致严重 hallucination

### 结论

视频输入 (C) 性价比最高: 成本几乎不变, 质量飞跃提升。建议 movement 描述任务全部用视频输入。

## 2026-04-20 Zone Partition 成本

**模型**: Gemini 2.5 Flash via OpenRouter

### 分区 pipeline (guided 模式)

| 步骤 | 每段调用次数 | 31段总调用 | 估算 tokens/调用 | 总成本 |
|---|---|---|---|---|
| 分区 (batch 6帧) | 1 | 31 | ~1,600 input + ~300 output | ~$0.013 |
| 标注 (6帧+zone info) | 1 | 31 | ~1,800 input + ~400 output | ~$0.016 |
| **合计** | **2** | **62** | | **~$0.029** |

### 对比: 旧版逐帧分区

| 方式 | API 调用数 | 估算总成本 |
|---|---|---|
| 逐帧分区 (旧) | 186 + 31 = 217 | ~$0.070 |
| batch 分区 (新) | 31 + 31 = 62 | ~$0.029 |
| **节省** | **155 次调用 (-71%)** | **~$0.041 (-59%)** |
